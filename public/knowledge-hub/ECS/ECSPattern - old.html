The Entity-Component System (ECS) pattern

The main purpose of this work is to describe the ECS pattern in an understandable way, and to elaborate on the principles that it supports. We do this by comparing how something would be modelled using ECS or a more classical Object-Oriented programming (OOP) approach. Finally, we will place this pattern within the field of Digital Twin Engineering.

ECS

ECS is a pattern that can be used to compensate for some of the downsides that a classical OOP approach has. OOP promotes principles such as encapsulation and inheritance. Encapsulation provides the benefit of keeping related code, both variables and methods, together. Code reusability is one of the main benefits that comes with inheritance. However, when systems are designed iteratively, the hierarchical inheritance structure might have changed frequently.

The ECS pattern has its roots in game development, where iterative development is common. In classical OOP, data/state and behavior of an object are combined into one class. In contrast, ECS it separates behavior from data/state. Furthermore, although inheritance is possibly in ECS (when combined with OOP), it promotes composition over inheritance to reduce the complexity of modelling new objects. This approach provides higher separation of concerns, allowing different development teams to more easily collaborate when iteratively developing a system[1], [2], [3]. An example of where inheritance can be problematic, would be the modelling of an object that would require the object to inherit from multiple classes. Using multiple inheritance is often inadvisable, as it increases complexity and might create ambiguity when the classes inherited from define the same variables or methods, but with different implementations.


In short, ECS has three major concepts: Entities, Components, and Systems. The figure shows some common examples for each concept, which can be commonly found within game design. In the next sections, we will explain the concepts in more detail.

Entities

An entity is nothing more than a container with a unique identifier that is composed of components. This implies that it would be sufficient to represent an entity by naught more than a unique integer, but often it is chosen to supplement this with more descriptive information on what the entity is meant to model [4]. Sometimes it is also chosen to assign family labels, or archetypes, to an entity depending on what sets of components it is composed of. This approach makes managing the entities easier [5].

Components

The term Component in ECS is quite different from the general definition of Component within the field of software engineering. Overall, we think that the term Attribute or Property would fit better. In ECS, an Entity is said to be composed of certain Components. Referring to it as attributes/properties would work, as this would imply that an entity can be composed of different attributes or properties, implying that said attributes/properties are being modeled for the specific entity instance. At any given time, such an attribute/property has a state, where the states of all attributes/properties define that state of the entity.

Geometry is a proper example, as this can be defined in different ways using different variables (e.g. length, width). Components could also be slightly more abstract, where a component itself serves as an identifying label to the entity that is composed from it (an example from game-design could be the differentiation between ally or foe).

Systems

The term System also does not align with the general definition of system used within software engineering.

Systems act on entities, computing state changes to relevant components. Systems act only upon certain entities, which depend on the components it has. As a very basic example, if gravity exists within the simulated world, a Gravity System could be responsible for calculating vectors based on the mass of entities. This system could act upon entities that are composed of a mass component. In this way, the behavior of an entity is determined by the systems that act on the entity, which is determined by the components it is composed of.

In naive implementations, Systems have to iterate over all entities to determine which should be processed. This can be expensive, especially in software systems where timing is important. The concept of entity families/archetype mitigates this, providing dynamic grouping of entities based on their component composition [4], [6]. It is possible for an entity to be part of multiple archetypes or families.

At design time, any system is unaware of the existence of entities, and only has a relation to components. The components, or more precisely their values, are the input to a system. The output can either be a description of the change to one or more component values, or the new state value of the component. Whether the output describes a change, or the new state value depends on whether other systems might also work on the same component, potentially also providing a change to the state value of a component (e.g. one system might calculate a force based on gravity, while another might do this based on other forces, resulting in a combined directional force).

Design vs runtime

Systems are designed to process certain components. As such, there is a strong dependency between the two at design time. The components an entity is composed of can change at runtime, and as such there is no direct relation between entity and system at design time.

At runtime, systems are required to search for entities that are composed of the set of components which they had a relation to at design time. The composition of entities can be changed freely at design time. This approach allows the addition or removal of behavior for entities during runtime, by changing their composition. Figure 4 illustrates the relationship between Entities, Systems, and Components at design and runtime.

It is also possible to change, add, or remove systems at runtime. Changing a system implementation effectively changes the behavior of all entities that are composed of components of the systems archetype. Adding a system at runtime could introduce new behavior to all entities, assuming the new system acts on existing components. Removing a system simply removes the behavior resulting from the execution of the removed system.

It is also possible to add new components at runtime. Existing (or new) entities could be composed with the new component. Without a system to act on this component, no behavioral change would be observed (assuming the component is new in the sense that it is independent of any existing system). It is possible to update a system (at runtime) to now require the new component, changing the behavior of all entities that are not composed of the new component.


Summary

In short, Systems model behavior. They transform the state of the input data, into an output state. This data is represented by Components. At runtime, sets of Components are given an identity, which results in the modelling of an Entity. Entities can modelled using any set of Components, and the Components of which and entity is composed can be changed during runtime. When the composition of components is changed, it might change the behavior of the entity, as different systems might act on it.

The strength of the ECS pattern lies in the ability to model entities using any combination of defined components. This is particularly useful with continuous development, or for system evolution, as new entities can be defined freely as long as a proper system orchestration is defined. As such, the modelling of entities is decoupled from their behavior, as behavior is defined by composing it of the right set of components.

The name of the ECS pattern is often considered as confusing, as it uses terms from computer and software engineering, with different meaning. Those that use the pattern are already familiar with the terms, and as such no name change is desired. However, we feel a different name would make it much easier to understand. For this we would propose the following: Entity-Property-Service (EPS). We feel the term entity is fitting. The term property fits much better for the term component. ECS systems might also be considered a type of service provider, providing some logical service to the system, particularly entities that have certain properties.
